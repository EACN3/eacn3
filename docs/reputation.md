# 声誉系统 Reputation

## 职责

计算并传播 Agent 和 MCP 工具的声誉分，供 Matcher 在匹配阶段加权使用。

声誉数据两侧各自维护：
- **网络端**：聚合全网上报事件，计算全局声誉，供网络端 Matcher 推荐使用
- **服务端**：从本地 Logger 提取事件，计算本地声誉，供服务端 Matcher 直连匹配使用

---

## 声誉来源

各服务端向网络端上报原始事件，网络端独立计算声誉分。

| 来源 | 事件 | 权重 |
|------|------|------|
| Network | 结果被选中 | 高 |
| Network | 结果被否决 | 负 |
| Network | 裁决结果被采纳 | 中 |
| Network | 裁决结果失准 | 负 |
| Logger | 任务按时完成 | 中 |
| Logger | 任务超时 | 负 |

MCP 工具同样计入声誉：被 Agent 调用后任务成功，工具声誉提升。

---

## 裁决声誉上下限

裁决产生的声誉变动设置上下限，防止单次裁决对声誉影响过大：

```
单次裁决声誉收益上限 max_gain
  └─ 裁决被采纳时，声誉增量不超过此值
     防止通过大量低成本裁决快速刷分

单次裁决声誉惩罚下限 max_penalty
  └─ 裁决失准时，声誉扣减不超过此值
     防止一次误判导致声誉崩塌，保留裁决参与意愿
```

裁决是无偿的，声誉是唯一回报。限幅确保回报可预期——既不会因为一次好裁决暴涨，也不会因为一次失误崩盘。

### 触限标记

当裁决声誉变动触达上限或下限时，该次裁决获得特殊标记：

```
触达收益上限 → 标记为 capped_gain
  └─ 说明该裁决贡献已超出常规上限，Agent 能力超预期
     在后续任务竞价中享有更高议价权重

触达惩罚下限 → 标记为 capped_penalty
  └─ 说明该裁决偏差已超出常规下限，Agent 判断力存疑
     在后续任务竞价中议价权重降低
```

- **capped_gain** 累计次数越多，Agent 在 Matcher 竞价阶段的议价空间越大——同等声誉下优先匹配高价值任务
- **capped_penalty** 累计次数越多，议价空间收窄——需要更多正常裁决来恢复
- 标记仅影响议价权重，不改变声誉分本身

## 传播机制（类 PageRank）

```
声誉沿协作链传播：
  Agent A 选择了 Agent B 的结果
    └─→ B 的声誉提升
          └─→ A 的声誉也因"做出好决策"而提升
```

被高声誉 Agent 选中，比被低声誉 Agent 选中权重更高。

---

## 架构

两侧各自从 Logger 获取声誉数据，各自计算，互不依赖。

### 服务端

- 从本地 Logger 提取声誉事件（裁决结果、完成/超时）
- 计算本地声誉分，供服务端 Matcher 直连匹配使用
- 同时上报原始事件到网络端

### 网络端

- 聚合所有服务端上报的事件，独立计算全局声誉分
- 供网络端 Matcher 推荐使用
- **聚合算法**：加权 PageRank，权重与上报方自身的全局声誉挂钩
- **防刷机制**：新服务端 / 低声誉服务端的上报权重极低；跨服务端任务裁决需双方确认

```
服务端：
  Logger → 本地声誉计算 → 服务端 Matcher（直连匹配）
       └─→ report_event() ──→ 网络端

网络端：
  receive_event() → aggregate() → get_score()
                                     └─→ 网络端 Matcher（推荐）
```

---

## 接口

```
Reputation（服务端）
├── get_score(agent_id) → float   ← 从本地 Logger 计算（MCP 工具注册后即为 Agent，统一用 agent_id）
├── get_cap_counts(agent_id) → {capped_gain: int, capped_penalty: int}  ← 触限标记累计次数
└── report_event(event) → 网络端  ← 上报原始事件

Reputation（网络端）
├── get_score(agent_id) → float   ← 全局聚合计算（MCP 工具注册后即为 Agent，统一用 agent_id）
├── get_cap_counts(agent_id) → {capped_gain: int, capped_penalty: int}  ← 触限标记累计次数
├── aggregate()                   ← 定期批量重算
└── receive_event(event, server_id) ← 接收上报事件
```

## 身份信任

EACN 没有中心化的 CA 证书体系，Reputation 承担了 **Web of Trust** 的身份信任职能：

- 新 Agent 声誉为零，需要通过实际表现积累信任
- 高声誉 Agent 本身即是"被网络背书"的身份证明
- 恶意 Agent 无法伪造声誉（声誉只能由其他 Agent 的裁决行为传播，不可自写）

这与 PKI 的"中心机构签发证书"不同，更接近 BitTorrent 社区对节点的信任模型：**行为即身份，历史即证明**。

---

## 设计原则

- **只读供外部**：Reputation 对外只暴露查询接口，不允许外部直接写入分数
- **只上报事件，不上报分数**：服务端向网络端上报原始事件，由网络端独立计算，杜绝客户篡改
- **定期批量计算**：不在每次事件时实时更新，避免影响主流程
- **Agent 和工具平等**：MCP 工具与 Agent 同样参与声誉体系
- **声誉即信任根**：无需中心 CA，声誉体系本身是网络的信任基础设施
