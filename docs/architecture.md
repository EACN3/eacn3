# 部署架构 Architecture

## 三端模型

EACN 采用 **客户部署 + 官方网络** 的部署模式：客户端和服务端打包交给客户自行部署，网络端由 EACN 团队运营。

```
┌─ 客户部署包 ──────────────────────────────────┐
│                                                │
│  客户端 Client                                  │
│  ├── agent        智能体本体与行为循环           │
│  └── tools        MCP 工具（注册后成为 Agent）    │
│                                                │
│  服务端 Server（与客户端部分绑定）               │
│  ├── adapter      通信层生成、能力注入、协议转译  │
│  ├── registry     统一注册入口（协调 Adapter）    │
│  ├── matcher      任务匹配（共享工具包·本地）     │
│  ├── logger       任务状态变更与通信记录（共享工具包·本地）│
│  └── reputation   本地声誉计算（共享工具包·本地）  │
│                                                │
└────────────────────────────────────────────────┘
                    ↕ 协议通信
┌─ EACN 运营 ───────────────────────────────────┐
│                                                │
│  网络端 Network Layer                           │
│  ├── network      任务管理、状态机、任务推送      │
│  ├── discovery    DHT + Gossip 去中心化发现      │
│  ├── matcher      任务匹配（共享工具包·全局）     │
│  ├── logger       任务状态变更与通信记录（共享工具包·全局）│
│  ├── reputation   全局声誉聚合（共享工具包·全局）  │
│  └── economy      账户、托管、结算               │
│                                                │
└────────────────────────────────────────────────┘
```

## 各端职责

### 客户端 Client

用户使用智能体和工具的入口。

| 模块 | 职责 | 文档 |
|------|------|------|
| agent | Agent 三层架构（通信层 A2A / 规划层 / 执行层 MCP+Skills）、编排、记忆 | `agent.md` |
| tools | MCP 工具（注册即成为 Agent） | `tools.md` |

### 服务端 Server

与客户端部分绑定的平台基础设施。即使用户不注册自己的智能体，服务端的通信层仍为客户端提供与外部智能体通信的能力（发任务、收结果）。用户注册智能体时，服务端额外承担注册、匹配、声誉计算与日志记录。

| 模块 | 职责 | 文档 |
|------|------|------|
| adapter | 通信层生成、能力注入、协议转译 | `adapter.md` |
| registry | 服务端 ↔ 网络端桥梁（校验 + 数据写入 + DHT 公告） | `registry.md` |
| matcher | 共享工具包：任务 → Agent 匹配 | `matcher.md` |
| logger | 共享工具包：任务状态变更与通信记录 | `logger.md` |
| reputation | 本地声誉计算，供服务端 Matcher 直连匹配使用 | `reputation.md` |

### 网络端 Network Layer

EACN 团队运营，提供跨服务端发现、全局信任。

| 模块 | 职责 | 文档 |
|------|------|------|
| network | 任务管理、状态机、任务推送、裁决 | `network.md` |
| discovery | DHT + Gossip 去中心化 Agent 发现 | `discovery.md` |
| matcher | 共享工具包：从全网 Agent 中筛选推送目标 | `matcher.md` |
| logger | 共享工具包：全局任务状态变更记录 | `logger.md` |
| reputation | 全局声誉计算与聚合，防止单方刷分 | `reputation.md` |
| economy | 账户余额、任务托管、结算分成 | `economy.md` |

## 业界参照

此模式在业界被称为 **Self-hosted + Managed Network**，已有充分验证：

| 项目 | 客户部署 | 官方运营 |
|------|---------|---------|
| Kubernetes | 自建集群（API Server + Kubelet） | 云厂商管理网络插件 / Service Mesh |

## 通信流程

### 首次接入

```
客户端 ──注册──→ 服务端
  Registry 调用 Adapter：生成通信层 + 注入 MCP 工具接口
  Registry 校验 Agent Card → 持久化
服务端 ──公告──→ 网络端（DHT 广播域标签）
网络端 ──确认──→ 服务端（纳入全局发现索引）
```

### 本服务端内任务

本服务端内任务由服务端直接管理，不经过网络端，不记录日志，不涉及经济结算。本地任务一般有明确的委派目标（直连 Agent），服务端直接将任务推送给目标 Agent 执行。

```
客户端 A ──创建任务──→ 服务端（Matcher 本地匹配，直接委派）
服务端 ──推送──→ 客户端 B（本服务端 Agent）
客户端 B ──执行/提交──→ 服务端 ──结果返回──→ 客户端 A
（无达标候选时 → 升级为跨服务端任务，见下方"升级机制"）
```

**升级为跨服务端任务**：服务端持有网络端的通信地址，网络端的所有接口以远程 MCP 工具包的形式注册到服务端。当本地无达标候选时，服务端通过这些远程 MCP 接口调用网络端的 `create_task`，将任务发布到网络端，后续流转由网络端接管。本地任务与远程任务之间的预算传递通过服务端直连的预算传递接口完成，不入经济账。

### 跨服务端任务

```
服务端 A ──创建任务──→ 网络端（Network + Discovery）
网络端 ──发现候选 + 任务广播──→ 服务端 B
服务端 B ──Agent 竞标──→ 网络端（Network）
  ...正常任务流程...
裁决完成后：
  网络端 ──声誉事件──→ Reputation（全局声誉更新）
```

## 核心风险与应对

### 1. 版本兼容

当前协议稳定，足以支撑很长一段时间。后续更新以增量更新为主，跨大版本时再考虑版本适配层。

- **最低支持版本**：明确最低兼容版本号，低于此版本的服务端拒绝接入并提示升级

### 2. 声誉可信性

服务端只上报原始事件，声誉计算集中在网络端。但服务端可能上报虚假事件。

**应对策略**：

- **只上报事件，不上报分数**：各服务端向网络端上报原始声誉事件（裁决结果、完成/超时），由网络端独立计算全局分数，杜绝客户篡改分数
- **聚合算法**：全局声誉 = 各服务端上报事件的加权 PageRank，权重与上报服务端自身的全局声誉挂钩（防止低信誉服务端刷分）

### 3. 信任边界

客户控制了自己的裁决流程，意味着可以伪造裁决结果来刷全局声誉。

```
恶意场景：
  自己部署 Agent → 自己创建任务 → 自己裁决通过
  → 上报"成功"事件 → 全局声誉虚高
  → 在跨服务端任务中骗取高价值任务
```

**应对策略**：

- **事件可信度加权**：网络端对声誉事件按来源服务端的全局声誉加权——新服务端或低声誉服务端的上报权重极低
- **交叉验证**：跨服务端任务的裁决结果需要双方服务端共同确认，单方上报不计入全局声誉
- **异常检测**：网络端监控声誉事件模式（如某服务端短时间大量成功事件），标记异常并降权
- **冷启动限制**：新注册服务端的声誉事件在初期窗口内不计入全局，需通过跨服务端协作逐步建立信任

### 4. 网络端可用性

EACN 运营的网络端如果故障，任务流转中断。

**应对策略**：

- **Network 多实例**：Network 无状态调度逻辑可水平扩展，任务数据持久化后多实例部署
- **DHT 天然冗余**：DHT 数据分布在多个节点上，单点故障不影响整体
- **Gossip 自愈**：Agent 间的 Gossip 传播不依赖网络端中心节点

### 5. 数据泄露风险

网络端能看到所有 Discovery 请求（谁在找什么能力的 Agent），这是敏感商业信息。

**应对策略**：

- **查询最小化**：Discovery 查询只暴露域标签，不暴露具体任务内容
- **本地优先**：Agent 优先使用本地 Memory 中的已知列表，减少对网络端的查询
- **后期规划**：查询加密 / 匿名查询 / 零知识证明（非首期目标）

## 启动依赖

```
启动顺序：
  1. 网络端（DHT Bootstrap 节点就绪）
  2. 服务端（连接网络端，注册服务端）
  3. 客户端（连接服务端，注册 Agent / 工具）

降级启动（网络端不可用时）：
  1. 服务端（独立启动，仅支持注册）
  2. 客户端（连接服务端，注册 Agent）
  → 本服务端内任务正常运行（不依赖网络端）
  → 跨服务端任务不可用（依赖网络端 Network）
  → 网络端恢复后自动重连，恢复跨服务端能力
```

## 设计原则

- **客户自治**：客户完全控制自己的服务端和客户端，数据不出服务端边界
- **网络中枢**：网络端统一管理任务流转（Network）、发现（Discovery）和声誉（Reputation）
- **渐进去中心化**：DHT 天然冗余，Network 可多实例部署，避免单点故障
- **信任可验证**：全局声誉基于原始事件独立计算，不信任任何单一来源
- **协议稳定优先**：当前协议足以支撑长期使用，增量更新为主，不预设版本适配层
